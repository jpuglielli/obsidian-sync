aSingle tree rooted at `/`. Everything is a file — devices, processes, kernel parameters.
## Layout
##### `/etc`
**"Editable Text Configuration."**: System configuration. 
- The networking stack lives here. `/etc/hosts` for local DNS overrides
- Service configs land here as well.
	- Basically any daemon looks here for its config. Examples:
		- `/etc/nginx/nginx.conf`, `/etc/ssh/sshd_config`, `/etc/kafka/server.properties`
- Authentication and access control:
	- E.g. `/etc/group` (group memberships), `/etc/sudoers` (who can sudo what — always edit with `visudo`).
- `/etc` is meant to be the _only_ directory you need to back up to recreate a system's configuration. 
	- Everything here is host-specific and text-editable. 
	- In containers, you typically mount config into `/etc` via ConfigMaps or volume mounts
##### `/var`
"Variable data": Things that can change at runtime.
- `/etc` is what you set up, `/var` is what accumulates at runtime
- `/var/log` is where you'll spend a lot of time. 
	- `syslog` or `journald` output, `auth.log` for SSH attempts, application logs.
	- When a service is misbehaving and you can't hit your observability stack, `tail -f /var/log/syslog` is your fallback.
- `/var/lib` holds persistent application state 
	- `/var/lib/postgresql` has your Postgres data directory
	- `/var/lib/docker` has all Docker images/containers/volumes 
	- `/var/lib/kubelet` on K8s nodes. 
- `/var/run` (often symlinked to `/run`) has PID files and Unix sockets. 
	- When you see `docker.sock` or a `.pid` file, it's here. 
	- These are ephemeral — gone on reboot.
- When a disk fills up and your service crashes, it's almost always `/var` that's full
	- `du -sh /var/*` 
##### `/tmp`
"Temporary Files"
- Cleared on reboot
- Anyone can write but only delete their own files. Applications use it for scratch space.
- `/tmp` is often mounted with `noexec` on hardened systems, which can break scripts that try to execute from there.
- In containers, `/tmp` is often a writable layer and can fill up your overlay filesystem if you're not careful with ephemeral storage limits in K8s.
##### `proc`
"**Proc**esses"
- This is a [[#Virtual Filesystems|virtual filesystem]]. Nothing gets generated on disk. Generated live by kernel, every process gets a dir at `/proc/[PID]/`
- How _all_ your monitoring tools work under the hood. Prometheus node_exporter? Reading `/proc`. `top`, `ps`, `netstat`? All just parsing `/proc`. When your monitoring is down, you can always go direct.
##### `/sys`
"**sys**tem": What hardware/subsystems exist? (devices, cgroups)
- Also a [[#Virtual Filesystems|virtual filesystem]], generated by kernel
- Won't interact very often directly
## Virtual Filesystems
Whereas a normal filesystem writes reads and writes actual bytes on a physical disk, a VFS is generated on the fly by the Kernel in memory, and has no physical backing.
- It uses the **same interface** — you can `ls`, `cat`, `echo >`, `stat` these paths like any real file.
	- The VFS (Virtual File System) layer in the kernel provides this uniform file interface, and different backends (ext4, procfs, sysfs, tmpfs) handle the actual implementation differently. 
	- Some go to disk, some just talk to the kernel.
**Example:**
When you run `cat /proc/cpuinfo`, the kernel isn't reading a file that someone wrote to disk — it's intercepting that read request and dynamically generating the output from its own internal data structures at that exact moment.
